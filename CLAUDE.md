## Project Summary

Calibrate Backend is a FastAPI REST API that wraps the `calibrate` CLI tool to orchestrate long-running evaluation and simulation jobs:

1. **STT/TTS evaluation** â€” benchmark speech providers against datasets
2. **LLM agent testing** â€” unit tests and multi-model benchmarks
3. **Voice/chat simulations** â€” simulated conversations between AI agents and personas across scenarios

The backend persists state in SQLite, stores artifacts in S3, and spawns `calibrate` CLI subprocesses to do the actual work. Authentication is Google OAuth or email/password, gated by JWTs.

## Testing Discipline (load-bearing)

Treat tests as part of the feature, not an afterthought. The rules are non-negotiable:

- **New feature â‡’ new tests.** Any new function, endpoint, router, helper, or branch added to `src/` must ship with tests that exercise it. Mirror the layout: `src/foo.py` â†’ `tests/test_foo.py`; a new router endpoint extends the matching `tests/test_routers_*.py` file. If no test file fits, create one alongside the existing ones â€” do not bury new behavior inside an unrelated test.
- **Modified feature â‡’ updated tests.** When you change behavior of existing code (bug fix, refactor, semantic change), update or add tests that pin the new behavior. If the existing test still passes after a real behavior change, the test is wrong â€” fix the test, don't silently lose the assertion.
- **Verify before claiming done â€” scoped only.** Before reporting any code change as working, run the tests that cover the changed code and confirm they pass. Run ONLY the scoped subset â€” the specific file (`uv run --group dev pytest tests/test_foo.py -q`) or `-k` selector that hits your change. **Do not run the full suite after every edit during iteration**; full-suite runs are slow and reserved for **once, immediately before pushing / opening the PR**. If a scoped run passes, that is sufficient evidence the change works â€” type checks and "the diff looks right" are not substitutes, but the full suite is overkill for individual edits.
- **No skipping or weakening tests to make them pass.** If a test fails after your change, the default assumption is the code is wrong, not the test. Only update an assertion when you have explicitly decided the old behavior was incorrect, and say so in the PR description.
- **Background-thread / subprocess code needs a test seam.** Job runners spawn `calibrate` and threads â€” tests stub `subprocess.Popen` / monkeypatch the runner registry rather than really launching CLI processes. Follow the patterns in `test_run_tasks.py`, `test_job_recovery.py`, and `test_routers_*.py`.

## Code Layout

All application code lives under `src/` and runs with `src/` as the working directory (see Dockerfile). Imports are flat (`from db import ...`, `from auth_utils import ...`) â€” there is no package namespace.

- `src/main.py` â€” FastAPI app, lifespan hooks (job recovery on startup), custom HTTP-Basic-auth'd `/docs` routes, Sentry/OTEL setup, router wiring
- `src/db.py` â€” single-file SQLite layer (~7000 lines). All schema DDL, migrations (wrapped in `try/except sqlite3.OperationalError` â€” see architecture.md gotcha on `DEFAULT CURRENT_TIMESTAMP`), and CRUD. Soft deletes (`deleted_at IS NULL` filter) everywhere.
- `src/utils.py` â€” S3 helpers, presigned URLs, `build_tool_configs()`, process-group kill helpers, job-queue registry (`register_job_starter`, `can_start_*_job`, `try_start_queued_*_job`), `TaskStatus` enum, `is_job_timed_out()`, Sentry capture helper
- `src/auth_utils.py` â€” JWT verification, `get_current_user_id` dependency, `require_superadmin`
- `src/dataset_utils.py` â€” `resolve_dataset_inputs()` / `inject_dataset_item_ids()` shared by STT and TTS routers
- `src/job_recovery.py` â€” on startup, kills orphaned process groups and restarts `in_progress` jobs
- `src/llm_judge.py` â€” `{{variable}}` template rendering, OpenRouter-based judge invocation, and `build_evaluator_cli_payload()` that shapes linked evaluators into the dict sent to the calibrate CLI (STT/TTS/LLM tests/simulations all share this).
- `src/annotation_eval_runner.py` â€” background worker for annotation-task evaluator runs; routes LLM-judge work through the calibrate CLI's STT `--eval-only` mode so judge logic isn't duplicated.
- `src/annotation_metrics.py` â€” human-vs-human and human-vs-evaluator agreement metrics for annotation tasks (pairwise mean agreement on shared rows; row-level overall has `evaluator_id=None`).
- `src/routers/` â€” one file per resource: `auth`, `user_limits`, `datasets`, `personas`, `scenarios`, `agents`, `tools`, `agent_tools`, `tests`, `agent_tests`, `stt`, `tts`, `simulations`, `evaluators`, `annotators`, `annotation_tasks`, `annotation_agreement`, `jobs`, `public`. Each router registers a job starter via `register_job_starter(...)` at module load so `try_start_queued_*_job` can resume queued work. `public.py` bypasses auth by design â€” its routes are excluded from the auth middleware so anyone with a valid `share_token` can view shared results.

## Architectural Load-Bearing Facts

These are the invariants most likely to trip up an edit. The architecture doc has the full explanation; this is the short form.

- **Three independent job queues**: eval (`stt-eval`, `tts-eval`), agent-test (`llm-unit-test`, `llm-benchmark`), simulation (`text`, `voice`). Each enforces both a global and per-user concurrency limit. Every mutation that completes/deletes/aborts a job must call `try_start_queued_*_job(...)` for that queue â€” otherwise queued work stalls.
- **User scoping is mandatory**: every DB call takes `user_id`, and routers return 404 (not 403) on access-denied to avoid leaking existence. `user_id` on `agent_test_jobs` / `simulation_jobs` is derived by JOIN through the parent entity.
- **SQLite stores UTC**; use `datetime.utcnow()` when comparing against `CURRENT_TIMESTAMP` columns. Never `datetime.now()`.
- **Subprocesses**: always start with `start_new_session=True`; store `pid`/`pgid` in job details; kill via `os.killpg` SIGTERMâ†’SIGKILL (catch both `ProcessLookupError` and `PermissionError` on macOS). Agent test jobs are the exception â€” they block on `process.wait()` and do not track PIDs because `agent_test_jobs` has no `details` column.
- **Polling loops that spawn `calibrate`**: redirect stdout/stderr to temp files (not pipes) to avoid buffer-full deadlocks; poll every 2s; only `update_job` when state actually changes (to preserve `updated_at` for the 5-minute timeout check); read intermediate outputs from disk.
- **Simulation abort**: check `_is_job_aborted(task_id)` at every layer (inside polling loop, inside intermediate-update writer, after loop exit, inside `run_simulation_task`, and all exception handlers). Missing any one re-opens a race where the monitoring thread overwrites the abort state.
- **STT/TTS intermediate results are disk-only** during `in_progress` â€” error/timeout handlers must call `_collect_intermediate_results()` / `_collect_tts_intermediate_results()` before writing failure to DB, or successful providers' data is lost. Timeout handlers must **merge** (not overwrite) with existing DB `success: true` entries.
- **Presigned URLs are generated on-the-fly** from stored S3 keys â€” never persist presigned URLs in the DB (they expire). Dataset item audio paths are stored as full `s3://bucket/key` URIs (exception to the bare-key convention); `_presign_audio_path()` handles parsing.
- **Benchmark model-to-folder matching is exact**, not substring. Substring matching previously caused silent cross-model contamination (e.g. `gpt-5.4` absorbing `gpt-5.4-mini`'s results). If the calibrate CLI adds a new folder naming convention, extend the candidate list in `_match_model_to_folder`.
- **`MAX_CONCURRENT_JOBS` default is 1** in `docker-compose.yml`. In practice jobs serialize; this matters when reasoning about the queue.
- **FastAPI upper bound** (`<0.122.0`) in `pyproject.toml` is load-bearing â€” it's constrained by `calibrate-agent` â†’ `pipecat-ai`. Don't loosen casually.
- **Dockerfile quirk**: `uv sync` installs into `.venv`, so any `RUN python ...` step must be `RUN uv run python ...` (see the `nltk.download('punkt_tab')` line).
- **Annotation-task bulk actions share a `select_all` + `q` contract.** `DELETE /annotation-tasks/{uuid}/items`, `POST /annotation-tasks/{uuid}/jobs`, and `POST /annotation-tasks/{uuid}/evaluator-runs` each accept `{item_ids: [...], select_all: bool=false, q: string?}`. `select_all=true` means "every non-deleted item in the task (optionally filtered by case-insensitive substring on `payload.name`, same rule as `_SummarySearch`)" and ignores `item_ids`; `select_all=false` (default) requires `item_ids` non-empty. Resolution funnels through `_resolve_target_item_ids` in [annotation_tasks.py](src/routers/annotation_tasks.py) â€” new bulk endpoints on the same router should reuse it instead of re-inventing the toggle. Note this is a breaking change from the prior evaluator-runs convention where `item_ids=None` meant "all items"; callers must now opt into `select_all=true` explicitly.
- **Annotation-task evaluator order is explicit, stored on the pivot.** `annotation_task_evaluators.position` drives every UI surface that lists a task's evaluators (task detail, `/evaluators` detail-shape endpoint, `/summary`, `/agreement`, per-item editor, annotator labelling job). New links append (`MAX(position)+1` among active rows); restore-on-relink also moves to the end. `reorder_evaluators_for_annotation_task` / `PUT /annotation-tasks/{uuid}/evaluators/order` re-numbers atomically and rejects sets that don't match the active linked set. Job snapshots are FROZEN: `create_annotation_job` copies the task's position-ordered list into `annotation_job_evaluators.position`, and later reordering on the task does NOT propagate to existing jobs. Every read path that lists task evaluators MUST `ORDER BY position ASC, id ASC` â€” anything that forgets it silently falls back to insertion order.
- **Annotation-job soft-delete cascades through reads, not rows.** `soft_delete_annotation_job` only flips `annotation_jobs.deleted_at`. The job's annotation rows are NOT touched. Every annotation read path (`get_annotations_for_task`, `get_annotations_for_item`, `get_annotations_for_slots`, `get_annotations_for_user`, `get_annotations_for_annotator_overlap_slots`, `get_annotated_item_ids`) JOINs on `annotation_jobs` and filters `j.deleted_at IS NULL`, so a soft-deleted job's annotations atomically vanish from list views, inter-annotator agreement, and the human columns of evaluator-run reports. This cascade is intentional and also outlasts the `include_deleted_items=True` reproducibility carve-out used for soft-deleted *items* in eval-run detail â€” that carve-out preserves item soft-deletes only, never job soft-deletes. Consequence: deleting an `annotation_jobs` row retroactively shrinks the `human_agreement` block of old evaluator runs that referenced it. New annotation reads must keep the same `j.deleted_at IS NULL` filter or they'll leak deleted-job labels into agreement.
- **Evaluators replace `metrics`**: the `metrics` table is frozen â€” new work goes to `evaluators` + `evaluator_versions`. `evaluators.owner_user_id IS NULL` â‡’ seeded default (visible to everyone, not editable). Custom evaluators duplicate-and-edit. Pivots (`simulation_evaluators`, `test_evaluators`) pin `evaluator_version_id` at link time so the "live version" API only affects *future* links â€” reruns of old tests keep using the old version. STT/TTS jobs store the same shape inside `details.evaluators` (no pivot table). `init_db()` seeds defaults with stable `slug`s, migrates legacy `metrics` rows into `evaluators` (marked via `source_metric_uuid`), and backfills LLM `tests` where `config.evaluation.type='response'` into a `test_evaluators` link against `default-llm-next-reply` with `variable_values={"criteria": <old text>}`. All three passes are idempotent â€” safe to rerun.
- **Evaluator shape**: `output_type` ("binary" | "rating") lives on the `evaluators` row â€” it's identity. The **rubric** (`output_config`, including scale values/labels/descriptions/colors) lives on each `evaluator_versions` row so prompt iterations carry their own pinned rubric. Pivots (`simulation_evaluators`, `test_evaluators`) pin `evaluator_version_id` at link time, so past runs stay reproducible even after the rubric is edited.
- **Evaluator type vs data type**: an evaluator carries **two independent classifying columns** on the `evaluators` row:
  - `evaluator_type` (`tts | stt | llm | conversation`, default `llm`) â€” semantic category. `tts` judges TTS output; `stt` judges one transcript in isolation; `llm` judges one response with conversation history; `conversation` judges a full conversation (used by simulations and `conversation`-type tests). The frontend filters per run context (`GET /evaluators?evaluator_type=stt` on the STT page, etc.). **This value was historically `simulation`**: `init_db()` runs an idempotent `UPDATE evaluators SET evaluator_type = 'conversation' WHERE evaluator_type = 'simulation'` migration. The same `simulation â†’ conversation` rename was applied in lockstep to the annotation-task `type` and the default-prompt `purpose` so all three axes share one vocabulary.
  - `data_type` (`text | audio`, default `text`) â€” the medium the judge actually consumes. **This is the only field that gates audio routing** in `invoke_evaluator()` (`data_type == "audio"` â‡’ audio bytes attached to the user message). It's also passed straight through to the calibrate CLI in `build_evaluator_cli_payload()`.
  In practice `tts â‡’ data_type=audio` and the rest are `text`, but the split is preserved so a user can in principle define, e.g., a TTS evaluator that judges a transcribed text instead of audio without reclassifying its category.
  The schema went through an intermediate phase where `data_type` was renamed to `evaluator_type`; `init_db()` now runs both `ALTER TABLE evaluators RENAME COLUMN data_type TO evaluator_type` **and** `ALTER TABLE evaluators ADD COLUMN data_type TEXT NOT NULL DEFAULT 'text'` (both try/except guarded â€” fresh DBs and DBs already on the new schema both no-op). Then it remaps legacy `evaluator_type` values (`audio â†’ tts`, `text â†’ llm`) and backfills `data_type = 'audio' WHERE evaluator_type = 'tts' AND data_type = 'text'` so partially-migrated DBs land in the right place. `_seed_default_evaluators` then snaps each seeded slug to its canonical `(evaluator_type, data_type)` pair. Default seeds: `default-tts-audio-quality` â†’ `(tts, audio)`; `default-stt-transcription` â†’ `(stt, text)`; the seeded `conversation` evaluators (`default-sim-goal-completion`, `default-sim-empathy-tone`, `default-sim-persona-adherence`) â†’ `(conversation, text)`; everything else (`default-llm-next-reply`, `default-faithfulness`, `default-helpfulness`, `default-safety`, `default-conciseness`, `default-instruction-following`) â†’ `(llm, text)`.
- **output_config shape** â€” same entry form for every output type: `{value, name, description?, color?}`. Binary = 2-entry scale with bool `value`; rating = Nâ‰Ą2-entry with numeric `value`. Adding a new `output_type` (categorical, ranking, â€¦) means a new JSON shape inside `output_config` â€” no schema change. `variables` also live per-version (they're defined by the prompt text).
- **Per-level rubric injection**: `_format_scale_rubric()` in [llm_judge.py](src/llm_judge.py) appends a `Rubric:\n  value (name): description\n...` block to the judge prompt whenever any scale entry has a `description`. Applied identically for the `/invoke` endpoint and for the calibrate CLI payload (baked into the rendered `system_prompt` of each evaluator entry). Entries without a description are silently skipped â€” mixed-fill is fine.
- **API keys (`/api-keys`)** use bcrypt; the raw key is returned exactly once on creation and the DB keeps only a `key_prefix` (first 12 chars) for lookup plus the bcrypt hash. `get_user_from_api_key` accepts either `X-API-Key: <key>` or `Authorization: Bearer <key>` (keys are prefixed `calib_` so they can coexist with JWTs in the same header).
- **Evaluator invocation (`POST /evaluators/{uuid}/invoke`)** is the only endpoint that uses API-key auth instead of JWT. It renders `{{variable}}` placeholders against the request's `variables` dict, sends to OpenRouter, and enforces JSON output shape (`{pass, reasoning}` for binary, `{value, reasoning}` for rating, `{winner, reasoning}` for `kind=side_by_side`).
- **Calibrate CLI evaluator handoff** â€” the calibrate CLI accepts a minimal evaluator definition: `{name, system_prompt, judge_model, type ("binary"|"rating"), scale_min?, scale_max?}`. The backend's stored extras (`evaluator_type`, `data_type`, `kind`, `output_config.scale.{name,description,color}`) are NOT sent â€” they're either inferred by calibrate from the parent flow (e.g. TTS implies audio) or get baked into the prompt before sending (e.g. per-level `description`s become a "Rubric:" block via `_format_scale_rubric`). For STT/TTS the payload goes to `--config <path>` as `{"evaluators": [...]}`. For simulations it lives at `config.evaluators` (replacing the old `evaluation_criteria` key). For LLM tests/benchmarks the helper `build_test_evaluators_payload` produces a deduped top-level `evaluators` list (one entry per unique evaluator UUID â€” names suffixed with `-{uuid8}` on collision) and per-test-case `config.test_cases[].evaluation.criteria = [{name, arguments?}]` references; the top-level prompts keep `{{variable}}` placeholders unrendered so calibrate substitutes per test case using each criterion's `arguments`. STT/TTS/simulation payloads instead pre-render variables (no per-row arguments mechanism).
- **Three test types, one execution path â€” all run the agent live.** `tests.type` is one of `response`, `tool_call`, `conversation` (validated in [tests.py](src/routers/tests.py); `response`/`tool_call` pin `llm` evaluators, `conversation` pins `conversation` evaluators). **Conversation tests require â‰Ą1 evaluator** â€” enforced on single create/update AND bulk â€” because they have no fallback judge (unlike `response`, which can synthesize the default LLM judge from a legacy string `criteria`); without one a run would build an empty calibrate config with nothing to judge. **All three run through the same `calibrate llm` command** via `_build_calibrate_config` â†’ `run_llm_test_task` ([agent_tests.py](src/routers/agent_tests.py)), and **all three invoke the agent** â€” `config.history` ends at the user turn the agent should answer; calibrate runs the agent to produce its reply, then dispatches per test case on `evaluation.type`: `response` â‡’ judge only the generated reply; `conversation` â‡’ append the reply and judge the FULL conversation with the conversation judge; `tool_call` â‡’ diff the generated tool calls. Because the wire contract is identical, `_build_calibrate_config` treats `response` and `conversation` the same: both contribute to the deduped top-level `evaluators` list and get per-test-case `evaluation.criteria = [{name, arguments?}]` refs (via `build_test_evaluators_payload`); the evaluator payload is the same shape regardless of evaluator_type. Consequences: runs CAN mix types in one `calibrate llm` call; conversation tests benchmark meaningfully (the agent generates per model, so judgments differ); and the agent-connection-verified guard applies **uniformly** to every run (conversation tests call the agent too). Output (`results.json`/`metrics.json`) and the read-path enrichment + `evaluators[]` block builder are shared unchanged â€” `output.response` carries the generated reply, `judge_results` are keyed by evaluator name with `match` (binary) / `score` (rating), and per-test-case `passed` is computed **by calibrate** (AND across evaluators: binary must `match`, rating must equal `scale_max`), not by the backend. **The immutable row `type` is the single source of truth for dispatch**, NOT the stored `config.evaluation.type`: `PUT /tests` keeps `type` fixed but accepts an arbitrary `config` dict, so the two can drift. `_build_calibrate_config` dispatches on the row `type` and **normalizes each test case's `evaluation.type` to it** before the CLI handoff, so a drifted config can never make calibrate judge the wrong way.
- **Shared list-endpoint deps live in [src/pagination.py](src/pagination.py).** Three pieces: `PaginationParams` (FastAPI dep for `?limit=&offset=`, default `limit=50`, max `1_000_000`, `offsetâ‰Ą0` â€” cap is deliberately huge so FE "export all" flows can pass `limit=total` without a multi-request loop; the cap is a sanity guard, not a payload budget), and two factories â€” `make_sort_params(sortable=[...], default=..., default_order="desc")` and `make_search_params(searchable=["payload.name", ...])`. Each factory returns a fresh `Depends`-compatible class baked with that endpoint's allowlist; the class exposes `.apply(items)` that returns a filtered/sorted copy. Per-endpoint usage: build the dep classes at module load (so FastAPI sees stable types and the factory's allowlist validation catches typos before request time), then `items = search.apply(items); items = sort.apply(items); page = items[offset:offset+limit]`. Sort's `apply()` defaults to `uuid` as a secondary key so identical-timestamp items don't shuffle across page boundaries (sqlite `CURRENT_TIMESTAMP` is second-resolution); pass `secondary_key="id"` for entities where bulk insertion is the common path (e.g. annotation items â€” `id` is the autoincrement and preserves insertion order, whereas `uuid` is random and would shuffle whole batches). Search standardizes on `?q=` (no per-endpoint name override) and case-insensitive substring; blank/whitespace is a no-op so FE inputs don't need clearing logic. Disallowed `sort_by` 422s at request time via manual `HTTPException` since `Literal` can't be built dynamically. Search/sort run **post-fetch in Python** â€” fine at current entity scale; push down into SQL inside the handler if a table outgrows that. Only `GET /annotation-tasks/{uuid}/summary` uses these today (paginated at the **item level** since each item explodes into â‰Ą1 row per `evaluator Ă— version`, with `evaluators[]`/`annotators[]` blocks stable across pages); the response inlines `pagination: {total, limit, offset}` rather than wrapping the whole composite in a generic envelope. Other list endpoints (`/agents`, `/tools`, â€¦) still return full lists â€” adopt the deps when adding pagination; don't invent a per-router shape. If you need a paginated-envelope wire shape (`{items, total, limit, offset}`) for a plain list endpoint, see commit `c21dd05` for an earlier `PaginatedResponse[T]` + `count_and_page` draft.
- **Default prompts API** â€” `GET /evaluators/default-prompt?purpose={llm|stt|tts|conversation}` returns the canonical default system prompt + suggested config for prefilling the create-evaluator form. The same prompts power the seeded LLM/STT/TTS evaluators (built via `_seed_from_purpose` in [db.py](src/db.py) so `DEFAULT_PROMPTS_BY_PURPOSE` is the single source of truth). The `conversation` purpose's prompt is a template with a literal `<ENTER EVALUATION CRITERIA HERE>` placeholder the user replaces directly (no `{{var}}`). Default judge models: `DEFAULT_TEXT_JUDGE_MODEL` for text/STT/conversation, `DEFAULT_AUDIO_JUDGE_MODEL` for TTS.

## Conventions

- New entities: UUID primary identifier (plus auto-increment `id`), `deleted_at` soft delete, `created_at`/`updated_at` timestamps, JSON `config` column for flexible fields, validated at the API layer via Pydantic.
- Many-to-many links use dedicated pivot tables with their own soft-delete column.
- When mutating child rows (e.g. `dataset_items`), bump the parent's `updated_at` in the same transaction.
- Schema migrations on existing tables: `ALTER TABLE ADD COLUMN` inside `init_db()`, wrapped in `try/except sqlite3.OperationalError: pass`. Use `DEFAULT NULL` â€” SQLite silently rejects non-constant defaults like `CURRENT_TIMESTAMP` in `ADD COLUMN`, and the `except` will swallow the failure.
- All job failures route through `capture_exception_to_sentry()` (marks as unhandled, flushes immediately â€” critical for background threads).
- CLI failure detection is two-layer: non-zero exit code **or** expected structured output missing. Never treat stderr content as a failure signal â€” the calibrate CLI emits benign cleanup tracebacks.
