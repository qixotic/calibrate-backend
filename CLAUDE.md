# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Workflow

- **Start every new task in a fresh worktree when you're on `main`.** Before touching anything, check the current branch. If it's `main`, do NOT work on it directly — create a new git worktree on its own branch for the task and do all the work there. Only skip this if the user has already put you on a task branch/worktree or explicitly says to work on the current branch.
- **Before any change or new feature, do this first — every time.** Whenever you're asked to make a change, add a feature, or modify anything:
  1. **Review the existing code thoroughly** for code/functionality that already does (or partly does) what's being asked — don't build on an empty assumption.
  2. **Make a plan** that splits the work into independent subtasks, to be executed in parallel by multiple weaker agents wherever the subtasks are disjoint (see the `parallelize` skill).
  3. **Identify code to reuse or repackage** into shared, reusable functions to support what needs to be built, rather than duplicating logic.
  4. **Prefer well-known, reliable libraries** over re-implementing functionality that already exists in one — unless the requirements genuinely demand a custom solution.
  5. **Share the plan and ask** about any choices that need to be made, laying out the tradeoffs, instead of making assumptions on the user's behalf.
- **When adding or editing an API endpoint, ask whether it should be public — unless it already is.** Any time you create a new route or change an existing one, check whether it already carries `tags=["Public API"]` (see the "Public API docs are tag-gated" fact). If it does, leave it public. If it does NOT, ask the user whether to expose it on the unauthenticated public API page (which also means wiring it to `get_org_jwt_or_api_key` so API-key clients can call it) or keep it JWT-only — lay out the tradeoff, don't decide silently. Never publish a route by adding the tag without explicit confirmation.
- **When adding or editing a route, follow the `api-writing-style` skill for its docs.** Every endpoint gets an explicit imperative `summary=`, a concise verb-first docstring, and `Field`/`Path`/`Query` descriptions on all params and request/response models. The full house style (verb vocabulary, shape hints, per-endpoint checklist) lives in [.cursor/skills/api-writing-style/SKILL.md](.cursor/skills/api-writing-style/SKILL.md) — read it before documenting routes. Enforced by [tests/test_api_docs_style.py](tests/test_api_docs_style.py) (also runnable via `uv run python scripts/check_api_docs_style.py`).
- **Before claiming any change that touched a rendered API doc string is done, run the `api-docs-audit` skill.** If the change added or edited a route `summary=`, a route docstring, a Pydantic model class docstring, or any `Field`/`Path`/`Query` `description=` (including factory-generated ones), audit the change with [.cursor/skills/api-docs-audit/SKILL.md](.cursor/skills/api-docs-audit/SKILL.md): it inventories every changed rendered string, verifies each against `api-writing-style` (one sub-agent per string, covering the judgment-call rules the mechanical gate can't), and publishes a field-level HTML report grouped by file then method. This is in addition to `tests/test_api_docs_style.py`, not a replacement — the test catches only the machine-checkable subset (em-dashes, semicolons, unit echoes); the audit catches explanatory parentheticals, per-X phrasing, null caveats, enum re-listing, and consistency. Fix every violation it surfaces (generalizing the fix across the surface) and re-run until the report is clean before reporting done.
- **When the Public API surface changes, update client overlays in the same PR — every time.** Adding, removing, or renaming a route tagged `Public API` is not done until **both** naming overlays are updated manually in that change: [fern/openapi-overrides.yml](fern/openapi-overrides.yml) (Fern Python SDK: `x-fern-sdk-group-name` / `x-fern-sdk-method-name`) and [openapi/overlay.yaml](openapi/overlay.yaml) (Speakeasy CLI: `x-speakeasy-group` / `x-speakeasy-name-override`, plus `x-speakeasy-mcp` for MCP tool metadata). Overlays are **not** generated from the spec — Fern and Speakeasy will still emit the new route without them, but with ugly auto-derived names. Run `uv run --group dev pytest tests/test_sdk_overrides.py -q` before claiming done; CI enforces exact parity with `_build_public_openapi()`. Removing a route: delete its entries from **both** files. Renaming a path or HTTP method: update both overlays to match. **Shipped SDK/CLI/MCP binaries auto-publish after production deploy when the public OpenAPI spec changes** (see [SETUP_SDK.md](SETUP_SDK.md)) — backend overlay fixes alone don't reach installed `pip`/`brew`/`npm` users until that deploy + publish cycle runs.
- **Write code comments per the `code-comments` skill.** Every `#` comment (and internal non-rendered docstring) must be standalone and timeless — written for a stranger reading the file cold, never a log of our conversation, the review round, or the edit that produced it. Comment the *why* (constraint, tradeoff, footgun), not the *what*; no "we"/"as discussed"/"previously… now"/"introduced here"; cross-reference by symbol name, not by story; one idea in the fewest words. The full rules and before/after examples live in [.cursor/skills/code-comments/SKILL.md](.cursor/skills/code-comments/SKILL.md). (Rendered API doc strings follow `api-writing-style` instead.)
- **Commit when the work is done — don't wait to be asked.** Once all the required changes for a task are made and the scoped tests pass, create the commit yourself. Use a focused message that explains the *why*. Do not push unless explicitly asked. The standard safety rules still apply: never amend published commits, never skip hooks, never `git add -A` blindly, never commit secrets.
- **Keep CLAUDE.md in sync with the app's mental model.** After making changes, ask yourself: does this commit shift a *high-level* understanding of the app — a load-bearing invariant, an architectural rule, a convention, a non-obvious gotcha, a new subsystem, or a contract between components? If yes, update CLAUDE.md in the **same commit** so future Claude sessions inherit the new mental model. Don't record routine fixes, file moves, renames, or anything obvious from reading the code — only changes a fresh reader would otherwise miss. When in doubt, prefer updating an existing bullet over adding a new one.

## Parallel execution (default)

For any multi-step or multi-file task, first write a short plan that splits the work into subtasks and marks which are independent. Run independent subtasks as parallel agents, each owning a disjoint set of files/sections so they never edit the same file at once. Keep dependent subtasks sequential. Show the plan before launching. See the `parallelize` skill for the full playbook.

## Project Summary

Calibrate Backend is a FastAPI REST API that wraps the `calibrate` CLI tool to orchestrate long-running evaluation and simulation jobs:

1. **STT/TTS evaluation** — benchmark speech providers against datasets
2. **LLM agent testing** — unit tests and multi-model benchmarks
3. **Voice/chat simulations** — simulated conversations between AI agents and personas across scenarios

The backend persists state in SQLite, stores artifacts in S3-compatible object storage (or local disk when `OBJECT_STORAGE_MODE=local` for development), and spawns `calibrate-agent` CLI subprocesses (PyPI package `calibrate-agent`, console script `calibrate-agent`) to do the actual work. Authentication is Google OAuth or email/password, gated by JWTs.

## Testing Discipline (load-bearing)

Treat tests as part of the feature, not an afterthought. The rules are non-negotiable:

- **New feature ⇒ new tests.** Any new function, endpoint, router, helper, or branch added to `src/` must ship with tests that exercise it. Mirror the layout: `src/foo.py` → `tests/test_foo.py`; a new router endpoint extends the matching `tests/test_routers_*.py` file. If no test file fits, create one alongside the existing ones — do not bury new behavior inside an unrelated test.
- **Modified feature ⇒ updated tests.** When you change behavior of existing code (bug fix, refactor, semantic change), update or add tests that pin the new behavior. If the existing test still passes after a real behavior change, the test is wrong — fix the test, don't silently lose the assertion.
- **Verify before claiming done — scoped only.** Before reporting any code change as working, run the tests that cover the changed code and confirm they pass. Run ONLY the scoped subset — the specific file (`uv run --group dev pytest tests/test_foo.py -q`) or `-k` selector that hits your change. **Do not run the full suite after every edit during iteration**; full-suite runs are slow and reserved for **once, immediately before pushing / opening the PR**. If a scoped run passes, that is sufficient evidence the change works — type checks and "the diff looks right" are not substitutes, but the full suite is overkill for individual edits.
- **No skipping or weakening tests to make them pass.** If a test fails after your change, the default assumption is the code is wrong, not the test. Only update an assertion when you have explicitly decided the old behavior was incorrect, and say so in the PR description.
- **Background-thread / subprocess code needs a test seam.** Job runners spawn `calibrate` and threads — tests stub `subprocess.Popen` / monkeypatch the runner registry rather than really launching CLI processes. Follow the patterns in `test_run_tasks.py`, `test_job_recovery.py`, and `test_routers_*.py`.

## Code Layout

All application code lives under `src/` and runs with `src/` as the working directory (see Dockerfile). Imports are flat (`from db import ...`, `from auth_utils import ...`) — there is no package namespace.

- `src/main.py` — FastAPI app, lifespan hooks (job recovery on startup), custom HTTP-Basic-auth'd `/docs` routes, Sentry/OTEL setup, router wiring
- `src/db.py` — single-file SQLite layer (~7000 lines). All schema DDL, migrations (wrapped in `try/except sqlite3.OperationalError` — see architecture.md gotcha on `DEFAULT CURRENT_TIMESTAMP`), and CRUD. Soft deletes (`deleted_at IS NULL` filter) everywhere.
- `src/utils.py` — S3 helpers, presigned URLs, `build_tool_configs()`, process-group kill helpers, job-queue registry (`register_job_starter`, `can_start_*_job`, `try_start_queued_*_job`), `TaskStatus` enum, `is_job_timed_out()`, Sentry capture helper
- `src/auth_utils.py` — JWT verification, `get_current_user_id` dependency, `require_superadmin`
- `src/org_scope.py` — shared org-ownership guards (`ensure_owned_agent`, `ensure_owned_evaluator`) used by routers that link evaluators; 404 on cross-org access
- `src/dataset_utils.py` — `resolve_dataset_inputs()` / `inject_dataset_item_ids()` shared by STT and TTS routers
- `src/job_recovery.py` — on startup, kills orphaned process groups and restarts `in_progress` jobs
- `src/llm_judge.py` — `{{variable}}` template rendering, OpenRouter-based judge invocation, and `build_evaluator_cli_payload()` that shapes linked evaluators into the dict sent to the calibrate CLI (STT/TTS/LLM tests/simulations all share this).
- `src/annotation_eval_runner.py` — background worker for annotation-task evaluator runs; routes LLM-judge work through the calibrate CLI's STT `--eval-only` mode so judge logic isn't duplicated.
- `src/annotation_metrics.py` — human-vs-human and human-vs-evaluator agreement metrics for annotation tasks (pairwise mean agreement on shared rows; row-level overall has `evaluator_id=None`).
- `src/routers/` — one file per resource: `auth`, `user_limits`, `datasets`, `personas`, `scenarios`, `agents`, `tools`, `agent_tools`, `tests`, `agent_tests`, `stt`, `tts`, `simulations`, `evaluators`, `annotators`, `annotation_tasks`, `annotation_agreement`, `jobs`, `public`. Each router registers a job starter via `register_job_starter(...)` at module load so `try_start_queued_*_job` can resume queued work. `public.py` bypasses auth by design — its routes are excluded from the auth middleware so anyone with a valid `share_token` can view shared results.

## Architectural Load-Bearing Facts

These are the invariants most likely to trip up an edit. The architecture doc has the full explanation; this is the short form.

- **Three independent job queues**: eval (`stt-eval`, `tts-eval`), agent-test (`llm-unit-test`, `llm-benchmark`), simulation (`text`, `voice`). Each enforces both a global and per-user concurrency limit. Every mutation that completes/deletes/aborts a job must call `try_start_queued_*_job(...)` for that queue — otherwise queued work stalls.
- **User scoping is mandatory**: every DB call takes `user_id`, and routers return 404 (not 403) on access-denied to avoid leaking existence. `user_id` on `agent_test_jobs` / `simulation_jobs` is derived by JOIN through the parent entity.
- **SQLite stores UTC**; use `datetime.utcnow()` when comparing against `CURRENT_TIMESTAMP` columns. Never `datetime.now()`.
- **Subprocesses**: always start with `start_new_session=True`; store `pid`/`pgid` in job details; kill via `os.killpg` SIGTERM→SIGKILL (catch both `ProcessLookupError` and `PermissionError` on macOS). Agent test jobs are the exception — they block on `process.wait()` and do not track PIDs because `agent_test_jobs` has no `details` column.
- **Polling loops that spawn `calibrate`**: redirect stdout/stderr to temp files (not pipes) to avoid buffer-full deadlocks; poll every 2s; only `update_job` when state actually changes (to preserve `updated_at` for the 5-minute timeout check); read intermediate outputs from disk.
- **Simulation abort**: check `_is_job_aborted(task_id)` at every layer (inside polling loop, inside intermediate-update writer, after loop exit, inside `run_simulation_task`, and all exception handlers). Missing any one re-opens a race where the monitoring thread overwrites the abort state.
- **STT/TTS intermediate results are disk-only** during `in_progress` — error/timeout handlers must call `_collect_intermediate_results()` / `_collect_tts_intermediate_results()` before writing failure to DB, or successful providers' data is lost. Timeout handlers must **merge** (not overwrite) with existing DB `success: true` entries.
- **Artifact storage mode**: production/default is `OBJECT_STORAGE_MODE=s3`, where `S3_OUTPUT_BUCKET` is required and presigned URLs are generated on-the-fly from stored S3 keys — never persist presigned URLs in the DB (they expire). Local development can set `OBJECT_STORAGE_MODE=local`; `get_s3_output_config()` returns the sentinel bucket `local-dev-artifacts`, `get_s3_client()` returns `None`, and the storage helpers route to `LOCAL_ARTIFACT_ROOT` instead of AWS — `upload_file_to_s3`/`download_file_from_s3`/`list_object_keys` all branch on `is_local_object_storage()`, so job runners must go through them rather than calling `s3.download_file`/`s3.get_paginator` on the (now `None`) client directly. `/local-artifacts/{key}` acts as the development upload/download URL (`LOCAL_ARTIFACT_BASE_URL` makes generated upload **and** download URLs absolute for the frontend — both `generate_presigned_upload_url` and `generate_presigned_download_url` read the env var, with no per-request base-URL fallback). Dataset item audio paths are still stored as full `s3://bucket/key` URIs (exception to the bare-key convention); `_presign_audio_path()` handles parsing.
- **Benchmark test scope mirrors the single run.** `POST /agent-tests/agent/{uuid}/benchmark` takes `{models: [...], test_uuids?: [...]}`. Omitted/null/empty `test_uuids` ⇒ all tests linked to the agent (the historical "benchmarks always run all tests" behavior). When provided, every uuid **must be linked to the agent** (404 with the offending ids listed otherwise) — stricter than `POST .../run`, whose `test_uuids` only checks the test *exists*, not that it's linked. Resolution preserves request order and de-dupes; the resolved list is snapshotted into the job `details.test_uuids`.
- **Benchmark model-to-folder matching is exact**, not substring. Substring matching previously caused silent cross-model contamination (e.g. `gpt-5.4` absorbing `gpt-5.4-mini`'s results). If the calibrate CLI adds a new folder naming convention, extend the candidate list in `_match_model_to_folder`.
- **`MAX_CONCURRENT_JOBS` default is 1** in `docker-compose.yml`. In practice jobs serialize; this matters when reasoning about the queue.
- **Two CLIs, two names.** Offline evals use PyPI **`calibrate-agent`** (console script **`calibrate-agent`**, hardcoded via `get_calibrate_agent_cli()` in [utils.py](src/utils.py)). The Fern-generated **cloud API client** is command **`calibrate`** ("Calibrate CLI") — see the Public API docs bullet. Job runners must spawn the offline binary only through `get_calibrate_agent_cli()`.
- **FastAPI upper bound** (`<0.122.0`) in `pyproject.toml` is load-bearing — it's constrained by `calibrate-agent` → `pipecat-ai`. Don't loosen casually.
- **Dockerfile quirk**: `uv sync` installs into `.venv`, so any `RUN python ...` step must be `RUN uv run python ...` (see the `nltk.download('punkt_tab')` line).
- **Annotation-task bulk actions share a `select_all` + `q` contract.** `DELETE /annotation-tasks/{uuid}/items`, `POST /annotation-tasks/{uuid}/jobs`, and `POST /annotation-tasks/{uuid}/evaluator-runs` each accept `{item_ids: [...], select_all: bool=false, q: string?}`. `select_all=true` means "every non-deleted item in the task (optionally filtered by case-insensitive substring on `payload.name`, same rule as `_SummarySearch`)" and ignores `item_ids`; `select_all=false` (default) requires `item_ids` non-empty. Resolution funnels through `_resolve_target_item_ids` in [annotation_tasks.py](src/routers/annotation_tasks.py) — new bulk endpoints on the same router should reuse it instead of re-inventing the toggle. Note this is a breaking change from the prior evaluator-runs convention where `item_ids=None` meant "all items"; callers must now opt into `select_all=true` explicitly.
- **Annotation-task evaluator order is explicit, stored on the pivot.** `annotation_task_evaluators.position` drives every UI surface that lists a task's evaluators (task detail, `/evaluators` detail-shape endpoint, `/summary`, `/agreement`, per-item editor, annotator labelling job). New links append (`MAX(position)+1` among active rows); restore-on-relink also moves to the end. `reorder_evaluators_for_annotation_task` / `PUT /annotation-tasks/{uuid}/evaluators/order` re-numbers atomically and rejects sets that don't match the active linked set. `set_evaluators_for_annotation_task` / `PUT /annotation-tasks/{uuid}/evaluators` is the superset: it reconciles membership (links new, unlinks omitted) AND sets `position` to the given list order in one atomic call (so it does not require the target to match the current set), returning `{linked, unlinked}`. It's the Public API surface for task↔evaluator association (the single-item link `POST` was demoted to JWT-only), accepts an API key, rejects duplicate ids (400), and validates the whole set before mutating. Job snapshots are FROZEN: `create_annotation_job` copies the task's position-ordered list into `annotation_job_evaluators.position`, and later reordering on the task does NOT propagate to existing jobs. Job creation can snapshot a **subset** of the task's linked evaluators: `POST /annotation-tasks/{uuid}/jobs` accepts an optional `evaluator_ids` (validated to be a subset of the task's live links; empty list or unknown id ⇒ 400) that `create_annotation_job` filters the snapshot down to — preserving the task's position order among the chosen ones; omit it (`None`) to snapshot all. The labelling form (`get_evaluators_for_job` → public job payload) shows exactly the snapshotted set, so a subset job shows only those evaluators. Every read path that lists task evaluators MUST `ORDER BY position ASC, id ASC` — anything that forgets it silently falls back to insertion order.
- **Agent↔evaluator links use an unordered pivot.** `agent_evaluators (agent_id, evaluator_id, created_at, deleted_at)` mirrors `agent_tests` / `test_evaluators` soft-delete semantics: `add_evaluator_to_agent` inserts, restores soft-deleted rows on re-link, and is idempotent on an already-active link (no 500); `delete_agent` cascades soft-delete to the pivot; agent duplicate copies active links. Public API: `GET /agents/{uuid}/evaluators` (paginated slim `EvaluatorResponse`) and `POST …/evaluators` (link one or more: `{evaluator_ids: [...]}`, additive — skips already-linked, returns `{linked, already_linked}`); JWT-only: `DELETE` unlink one (removal stays off the key API). There is no set-replace PUT — additions go through `POST`, removals through `DELETE`. The `POST` accepts an API key, validates each id as exactly 36 chars (`EvaluatorUuid` in [utils.py](src/utils.py) — malformed ids 422 before DB lookup) and validates the whole set up front (bad id 404, nothing linked). Ownership checks go through [org_scope.py](src/org_scope.py) (`ensure_owned_agent` / `ensure_owned_evaluator`, the latter allowing seeded defaults with `org_uuid IS NULL`). **One-time deploy backfill** — when the pivot first ships, `init_db()` copies every evaluator reachable via `agent_tests` → `test_evaluators` onto the agent (`INSERT OR IGNORE … SELECT DISTINCT`). Gated by a `_schema_migrations` flag (`agent_evaluators_from_test_evaluators_v1`), not table existence — so a crash between CREATE and backfill retries on the next `init_db()`, and later runs never undo manual unlinks. Row count for ops logging uses `SELECT changes()` immediately after the insert (not `cursor.rowcount`, which is unreliable on `INSERT…SELECT`).
- **Annotation-job soft-delete cascades through reads, not rows.** `soft_delete_annotation_job` only flips `annotation_jobs.deleted_at`. The job's annotation rows are NOT touched. Every annotation read path (`get_annotations_for_task`, `get_annotations_for_item`, `get_annotations_for_slots`, `get_annotations_for_user`, `get_annotations_for_annotator_overlap_slots`, `get_annotated_item_ids`) JOINs on `annotation_jobs` and filters `j.deleted_at IS NULL`, so a soft-deleted job's annotations atomically vanish from list views, inter-annotator agreement, and the human columns of evaluator-run reports. This cascade is intentional and also outlasts the `include_deleted_items=True` reproducibility carve-out used for soft-deleted *items* in eval-run detail — that carve-out preserves item soft-deletes only, never job soft-deletes. Consequence: deleting an `annotation_jobs` row retroactively shrinks the `human_agreement` block of old evaluator runs that referenced it. New annotation reads must keep the same `j.deleted_at IS NULL` filter or they'll leak deleted-job labels into agreement.
- **Evaluators replace `metrics`**: the `metrics` table is frozen — new work goes to `evaluators` + `evaluator_versions`. `evaluators.owner_user_id IS NULL` ⇒ seeded default (visible to everyone, not editable). Custom evaluators duplicate-and-edit. **All four flows (LLM tests, simulations, STT, TTS) resolve the evaluator's *current* live version at run time** — the pivots/snapshots still store a pinned `evaluator_version_id` (FK NOT NULL / snapshot carrier) but it's only a fallback for the degenerate deleted/no-live-version case. **LLM tests and simulations resolve live via a JOIN-through-live**: `get_evaluators_for_test` and `get_evaluators_for_simulation` JOIN the version columns through the evaluator's `live_version_id` (`COALESCE(e.live_version_id, te.evaluator_version_id)` / `COALESCE(e.live_version_id, se.evaluator_version_id)`, surfacing `ev.uuid` as the version id — same trick), so every run uses the latest evaluator prompt/rubric and editing an evaluator changes what already-linked tests/simulations use. **STT/TTS resolve live by re-hydrating at run time**: `_refresh_evaluators_to_live` (in [stt.py](src/routers/stt.py) / [tts.py](src/routers/tts.py)) re-reads each evaluator's `live_version_id` when the job executes and writes the refreshed snapshot back into job details before building the calibrate payload (submit-time `details.evaluators` snapshot kept as fallback if the evaluator was deleted/has no live version). Per-link/per-job `variable_values` stay pinned everywhere (they're config, not part of the version). Every finished run of all four flows still renders reproducibly because each snapshots the live-at-run-time version into its job (see Evaluator shape bullet). STT/TTS jobs store the same shape inside `details.evaluators` (no pivot table). `init_db()` seeds defaults with stable `slug`s, migrates legacy `metrics` rows into `evaluators` (marked via `source_metric_uuid`), and backfills LLM `tests` where `config.evaluation.type='response'` into a `test_evaluators` link against `default-llm-next-reply` with `variable_values={"criteria": <old text>}`. All three passes are idempotent — safe to rerun.
- **Evaluator shape**: `output_type` ("binary" | "rating") lives on the `evaluators` row — it's identity. The **rubric** (`output_config`, including scale values/labels/descriptions/colors) lives on each `evaluator_versions` row so prompt iterations carry their own pinned rubric. **LLM tests, simulations, AND STT/TTS all resolve the live version at run time** (see Evaluators-replace-metrics bullet — tests/simulations via the pivot JOIN-through-live, STT/TTS via run-time re-hydration), but each *finished* run snapshots the live-at-run-time rubric into its job (e.g. `evaluators_by_test_id` in [agent_tests.py](src/routers/agent_tests.py) for tests; `_snapshot_evaluators_for_job_details` for simulations; the refreshed `details.evaluators` for STT/TTS — all stored on the job and re-read via [utils.py](src/utils.py)), so the completed run's results still render against the exact rubric it used even after the evaluator is edited again. **Variables (`{{placeholder}}` names) are immutable across versions**: `create_evaluator_version` rejects (ValueError ⇒ 400) any new version whose variable *name set* differs from the prior version — only descriptions/defaults may change, and version 1 is unconstrained. This is load-bearing for the live-resolution path above for every flow that keeps pinned `variable_values` (LLM tests and simulations): the pivot pins `variable_values` but runs the live prompt, so if a later version could rename/add/remove a variable the pinned values would no longer fill the live placeholders, silently sending a half-rendered prompt to the judge. The frontend already forbids editing variables post-creation; this is the backend enforcement.
- **Evaluator type vs data type**: an evaluator carries **two independent classifying columns** on the `evaluators` row:
  - `evaluator_type` (`tts | stt | llm | llm-general | conversation`, default `llm`) — semantic category. `tts` judges TTS output; `stt` judges one transcript in isolation; `llm` judges one response with conversation history; `llm-general` judges a standalone, **non-conversational** `input → output` pair (generic LLM use cases); `conversation` judges a full conversation (used by simulations and `conversation`-type tests). The frontend filters per run context (`GET /evaluators?evaluator_type=stt` on the STT page, etc.). **This value was historically `simulation`**: `init_db()` runs an idempotent `UPDATE evaluators SET evaluator_type = 'conversation' WHERE evaluator_type = 'simulation'` migration. The same `simulation → conversation` rename was applied in lockstep to the annotation-task `type` and the default-prompt `purpose` so all three axes share one vocabulary — which `llm-general` also spans (evaluator_type, annotation-task `type` in `ANNOTATION_TASK_TYPES`, and default-prompt `purpose`). The axes are NOT identical: **`tts` is a valid `evaluator_type` and default-prompt `purpose` but NOT an annotation-task `type`** — `ANNOTATION_TASK_TYPES` is `stt | llm | llm-general | conversation` (there are no TTS labelling tasks; the FE create dialog also excludes it). Since that set now equals `SUPPORTED_EVAL_TASK_TYPES`, the unsupported-type 400 guard in the evaluator-runs endpoint is defensive-only and unreachable through a real task. For the annotation **eval-runner**, `llm-general` is in `SUPPORTED_EVAL_TASK_TYPES` and uses the dedicated **`calibrate general`** command (non-conversational `input → output` judge — no agent, no conversation, no `--eval-only` flag): `_build_llm_general_dataset` emits flat `{id, input, output, arguments?}` rows; the config is the **unrendered** evaluator payload (`build_evaluator_cli_payload_unrendered`, same as the `llm` task type) so `{{placeholder}}`s survive for per-row substitution. Per-item `evaluator_variables` use the **same payload contract as the `llm` task type** (`{evaluator_uuid: {var: value}}`) and are resolved per-evaluator the same way (`_criteria_refs_for_item`); `calibrate general` takes them as a per-row `arguments` object **keyed by evaluator name** (it renders each evaluator's prompt against `arguments[ev_name]`), exactly mirroring `llm`'s per-criteria `arguments` — each evaluator gets its own box, so two evaluators sharing a `{{var}}` name never collide. Item payload is `{input, output, evaluator_variables?}` (vs the `llm` task type's `{chat_history, agent_response, evaluator_variables?}`). Output is `results.csv` (same column-map shape as STT: `id` + `<name>`/`<name>_reasoning` per evaluator, mapped via `config.json`'s `evaluators_map`), so `_parse_results_general` delegates to the STT CSV parser verbatim.
  - `data_type` (`text | audio`, default `text`) — the medium the judge actually consumes. **This is the only field that gates audio routing** in `invoke_evaluator()` (`data_type == "audio"` ⇒ audio bytes attached to the user message). It's also passed straight through to the calibrate CLI in `build_evaluator_cli_payload()`.
  In practice `tts ⇒ data_type=audio` and the rest are `text`, but the split is preserved so a user can in principle define, e.g., a TTS evaluator that judges a transcribed text instead of audio without reclassifying its category.
  The schema went through an intermediate phase where `data_type` was renamed to `evaluator_type`; `init_db()` now runs both `ALTER TABLE evaluators RENAME COLUMN data_type TO evaluator_type` **and** `ALTER TABLE evaluators ADD COLUMN data_type TEXT NOT NULL DEFAULT 'text'` (both try/except guarded — fresh DBs and DBs already on the new schema both no-op). Then it remaps legacy `evaluator_type` values (`audio → tts`, `text → llm`) and backfills `data_type = 'audio' WHERE evaluator_type = 'tts' AND data_type = 'text'` so partially-migrated DBs land in the right place. `_seed_default_evaluators` then snaps each seeded slug to its canonical `(evaluator_type, data_type)` pair. Default seeds: `default-tts-audio-quality` → `(tts, audio)`; `default-stt-transcription` → `(stt, text)`; the seeded `conversation` evaluators (`default-sim-goal-completion`, `default-sim-empathy-tone`, `default-sim-persona-adherence`) → `(conversation, text)`; everything else (`default-llm-next-reply`, `default-faithfulness`, `default-helpfulness`, `default-safety`, `default-conciseness`, `default-instruction-following`) → `(llm, text)`.
- **output_config shape** — same entry form for every output type: `{value, name, description?, color?}`. Binary = 2-entry scale with bool `value`; rating = N≥2-entry with numeric `value`. Adding a new `output_type` (categorical, ranking, …) means a new JSON shape inside `output_config` — no schema change. `variables` also live per-version (they're defined by the prompt text).
- **Per-level rubric injection**: `_format_scale_rubric()` in [llm_judge.py](src/llm_judge.py) appends a `Rubric:\n  value (name): description\n...` block to the judge prompt whenever any scale entry has a `description`. Applied identically for the `/invoke` endpoint and for the calibrate CLI payload (baked into the rendered `system_prompt` of each evaluator entry). Entries without a description are silently skipped — mixed-fill is fine.
- **API keys (`/api-keys`)** are credentials for programmatic API access, **scoped to one org**. The table stores a bcrypt `key_hash` plus the first-12-char `key_prefix` for candidate lookup; the raw `sk_…` key is returned exactly once at creation. After that, reads only ever expose `last_four` (the last 4 chars, stored in `key_last_four`) and a ready-to-render `masked_key` (`sk_••••<last_four>`) — never the prefix or hash. CRUD (`POST`/`GET`/`DELETE /api-keys`) is JWT-authed via `get_current_org`, so a key is minted in — and bound to — the caller's active org with the creator stamped as `owner_user_id` (revoke = soft delete). On the *consumption* side, `auth_utils.get_org_jwt_or_api_key` is a **required**-auth FastAPI dep that resolves an `OrgContext` from **either** a JWT (frontend) **or** an API key (any programmatic client): it accepts `X-API-Key: sk_…` or `Authorization: Bearer sk_…` (the `sk_` prefix lets keys coexist with JWTs in the same header), the key wins when present (owner role), it 401s on an invalid/revoked key, and otherwise falls back to `get_current_org` (401 if no JWT either). It's wired into the agent-test **run/poll** endpoints (`POST /agent-tests/agent/{uuid}/run`, `GET /agent-tests/run/{task_id}`) so both the FE and programmatic clients hit the same endpoints; ownership is enforced via `agent.org_uuid == ctx.org_uuid` (404 on mismatch). Benchmark + the rest of the agent-tests router stay JWT-only (`get_current_org`). One reference consumer is the GitHub Action in the separate `artpark/calibrate-github-action` repo (it hits `/agent-tests/agent/{uuid}/run` + `/agent-tests/run/{task_id}` with an `sk_` key).

- **Agent-test run-LIST endpoints are a lightweight index, not detail.** `GET /agent-tests/agent/{uuid}/runs` (public, MCP `list-agent-test-runs`) and `GET /agent-tests/runs` (JWT global) return `AgentTestRunListItem`s carrying only flat per-case `{name, passed}` (`TestRunCaseSummary` — the nested `test_case.name` is flattened up onto `name`), flat per-model scalar summaries (`ModelRunSummary` — no nested `test_results`), and run-level aggregates. They deliberately DROP the heavy detail for each case (agent `output`, `judge_results`, `reasoning`, the `test_case` body, per-case latency/cost), the per-model `test_results`, `leaderboard_summary`, and the top-level `evaluators` rubric block. Both frontends re-fetch that detail by task id when a run is opened (`GET /agent-tests/run/{task_id}` / `/benchmark/{task_id}`); the list only needs the slim fields for its pass/fail pills + display name. `_build_agent_test_run_item_fields` slims via `_slim_test_results`/`_slim_model_results` and skips evaluator enrichment entirely. This exists because the full-detail list blew up the payload (many runs × full detail for each case × each model) and closed the MCP stdio connection. Keep list items slim — put any new detail for a case or model on the DETAIL endpoints, never back on the list.
- **Agent-test runners share one launch path.** `_launch_agent_test_run(agent, tests, s3_bucket)` snapshots the calibrate config + per-test evaluator metadata, creates the `llm-unit-test` job, and starts/queues the thread (returns `(task_id, status)`). Two endpoints use it: `POST /agent-tests/agent/{uuid}/run` (one agent, returns `AgentTestRunCreateResponse` — `task_id` + `status` only) and the batch `POST /agent-tests/run` (returns `{runs: [{agent_name, agent_uuid, task_id, status}], skipped: [...]}`). The batch endpoint's optional body drives scope: `{agent_names: [...]}` runs just those agents; omitted/null/empty runs **every agent in your workspace**. When names are given they're **validated up front** — any unknown name 404s with `{not_found: [...]}` and creates NO jobs. It fans out via `_run_tests_for_agents`, **one job per agent**, skipping-and-reporting agents with no linked tests or an unverified connection (`skipped` reason `no_linked_tests | connection_not_verified`) rather than failing the batch. The single-agent endpoint instead **400s** on those same conditions (`_agent_connection_unverified` is the shared guard). All launched jobs are subject to the normal per-org concurrency queue, so over-limit jobs come back `status="queued"`.
- **Evaluator invocation (`POST /evaluators/{uuid}/invoke`)** is JWT-authed (`get_current_user_id`), like the rest of the API. It renders `{{variable}}` placeholders against the request's `variables` dict, sends to OpenRouter, and enforces JSON output shape (`{pass, reasoning}` for binary, `{value, reasoning}` for rating, `{winner, reasoning}` for `kind=side_by_side`).
- **Calibrate CLI evaluator handoff** — the calibrate CLI accepts a minimal evaluator definition: `{name, system_prompt, judge_model, type ("binary"|"rating"), scale_min?, scale_max?}`. The backend's stored extras (`evaluator_type`, `data_type`, `kind`, `output_config.scale.{name,description,color}`) are NOT sent — they're either inferred by calibrate from the parent flow (e.g. TTS implies audio) or get baked into the prompt before sending (e.g. per-level `description`s become a "Rubric:" block via `_format_scale_rubric`). For STT/TTS the payload goes to `--config <path>` as `{"evaluators": [...]}`. For simulations it lives at `config.evaluators` (replacing the old `evaluation_criteria` key). For LLM tests/benchmarks the helper `build_test_evaluators_payload` produces a deduped top-level `evaluators` list (one entry per unique evaluator UUID — names suffixed with `-{uuid8}` on collision) and per-test-case `config.test_cases[].evaluation.criteria = [{name, arguments?}]` references; the top-level prompts keep `{{variable}}` placeholders unrendered so calibrate substitutes per test case using each criterion's `arguments`. STT/TTS/simulation payloads instead pre-render variables (no per-row arguments mechanism).
- **Three test types, one execution path — all run the agent live.** `tests.type` is one of `response`, `tool_call`, `conversation` (validated in [tests.py](src/routers/tests.py); `response`/`tool_call` pin `llm` evaluators, `conversation` pins `conversation` evaluators). **Conversation tests require ≥1 evaluator** — enforced on single create/update AND bulk — because they have no fallback judge (unlike `response`, which can synthesize the default LLM judge from a legacy string `criteria`); without one a run would build an empty calibrate config with nothing to judge. **All three run through the same `calibrate llm` command** via `_build_calibrate_config` → `run_llm_test_task` ([agent_tests.py](src/routers/agent_tests.py)), and **all three invoke the agent** — `config.history` ends at the user turn the agent should answer; calibrate runs the agent to produce its reply, then dispatches per test case on `evaluation.type`: `response` ⇒ judge only the generated reply; `conversation` ⇒ append the reply and judge the FULL conversation with the conversation judge; `tool_call` ⇒ diff the generated tool calls. Because the wire contract is identical, `_build_calibrate_config` treats `response` and `conversation` the same: both contribute to the deduped top-level `evaluators` list and get per-test-case `evaluation.criteria = [{name, arguments?}]` refs (via `build_test_evaluators_payload`); the evaluator payload is the same shape regardless of evaluator_type. Consequences: runs CAN mix types in one `calibrate llm` call; conversation tests benchmark meaningfully (the agent generates per model, so judgments differ); and the agent-connection-verified guard applies **uniformly** to every run (conversation tests call the agent too). Output (`results.json`/`metrics.json`) and the read-path enrichment + `evaluators[]` block builder are shared unchanged — `output.response` carries the generated reply, `judge_results` are keyed by evaluator name with `match` (binary) / `score` (rating), and per-test-case `passed` is computed **by calibrate** (AND across evaluators: binary must `match`, rating must equal `scale_max`), not by the backend. **Both runners store calibrate's RAW parsed rows at write time — never validate `judge_results` through a Pydantic model there.** calibrate emits `judge_results` as a *dict* keyed by evaluator name, but `TestCaseResult.judge_results` is typed `List[JudgeResult]`; the dict→list conversion happens only at READ time in `_enrich_test_results_with_evaluators` / `_enrich_model_results_with_evaluators`. `run_llm_test_task` (and the benchmark intermediate writer) already store plain dicts; `run_benchmark_task` must do the same — it builds plain dicts for `model_results` and lets the read endpoint reshape them (constructing a `ModelResult(...)` at write time raised `ValidationError` for any `response`/`conversation` test, since `tool_call` has `judge_results=None` and slipped through). The `BenchmarkStatusResponse`/`ModelResult` validation still runs — but at read time, after enrichment. **Perf metrics (latency + cost + total_tokens)**: calibrate reports per-run perf metrics for live runs. **Per-case location differs by depth (this is a deliberate calibrate asymmetry — do not "fix" it):** latency is **top-level** `result["latency_ms"]` (float ms — calibrate rounds internal-model latency, but external agent tests self-report it verbatim and may send a fraction, so `TestCaseResult.latency_ms` is `Optional[float]`); cost is **nested** `result["output"]["cost"]` (float USD). The aggregated `metrics.json` blocks: `latency_ms` is `{p50,p95,p99,count}` (calibrate switched latency to **percentiles** — `mean/min/max/std/values` are gone; `p50` is the closest equivalent to the old `mean`), while `cost` and `total_tokens` stay `{mean,min,max,count}` (`count` = how many cases actually reported a value, never zero-filled). The same percentile switch applies to the **STT/TTS `ttfb`** block (now `{p50,p95,p99,count}`) and the leaderboards (TTS `ttfb` column → `ttfb_p50/p95/p99`; LLM `latency_ms` column → `latency_p50/p95/p99`). The backend stays unaffected because it treats all these as opaque pass-through: latency/ttfb aggregates flow through as `Dict[str, Any]`, and the leaderboard readers (`read_leaderboard_xlsx`, `_read_leaderboard_csv`) are column-name-agnostic — so adapting to the percentile shape / renamed columns is the **frontend's** job. (Old pre-change result folders still hold `ttfb.mean`/`latency_ms.mean`; re-running a leaderboard over them silently drops the latency columns since the new reader looks for `p50` — re-run the eval or accept the missing column.) The backend reads aggregates **straight from calibrate's `metrics.json` (it does NOT recompute them from rows)** — if calibrate omits a block the aggregate is null even when per-case rows have values (that's a calibrate bug to fix CLI-side, not a backend fallback). For agent-connection (external) tests the agent self-reports via an optional `metrics: {cost, latency_ms, prompt_tokens, completion_tokens}` dict in its JSON reply (missing/malformed → skipped, not errored). `--provider openai` reports no cost (route via OpenRouter for cost). The backend threads them through: `_parse_agent_test_results` lifts `r["latency_ms"]` AND `output.cost` up onto each `TestCaseResult` (per-case; there is NO per-case `total_tokens` — it's aggregate-only); unit-test runs store the aggregates at `results.{latency_ms,cost,total_tokens}` (on `TestRunStatusResponse`/list items/public run); benchmark runs store them per-model on `ModelResult.{latency_ms,cost,total_tokens}` so the FE can compare models on all three (the leaderboard CSV carries `latency_ms`/`cost` mean columns via generic dict pass-through when calibrate writes the leaderboard). **All are omitted for eval-only runs / no-cost providers / when calibrate doesn't write the block** — every field is `Optional`/`None`-defaulting, so treat them as optional everywhere. **The immutable row `type` is the single source of truth for dispatch**, NOT the stored `config.evaluation.type`: `PUT /tests` keeps `type` fixed but accepts an arbitrary `config` dict, so the two can drift. `_build_calibrate_config` dispatches on the row `type` and **normalizes each test case's `evaluation.type` to it** before the CLI handoff, so a drifted config can never make calibrate judge the wrong way.
- **Shared list-endpoint deps live in [src/pagination.py](src/pagination.py).** Pieces: `PaginationParams` (FastAPI dep for `?limit=&offset=`, default `limit=50`, max `1_000_000`, `offset≥0` — cap is deliberately huge so FE "export all" flows can pass `limit=total` without a multi-request loop; the cap is a sanity guard, not a payload budget); `OptionalPaginationParams` (same but `limit` **defaults to `None` = no limit** — omitting the params returns every item) paired with the `PaginatedResponse[T]` generic envelope model (`{items, total, limit, offset}`, `total` = **pre-slice** count) and helpers `paginate(items, pagination)` (slice → envelope dict in one call), `count_and_page(items, pagination) → (page, total)` + `page_envelope(page, total, pagination)` (for endpoints that transform each item before wrapping — e.g. the slim-model projections `_to_agent_summary`/`to_test_list_response`/`_evaluator_response`, run only over the returned page); and three factories — `make_sort_params(sortable=[...], default=..., default_order="desc")`, `make_search_params(searchable=["payload.name", ...])`, and `make_projection_params(heavy_fields=[...])`. Each factory returns a fresh `Depends`-compatible class baked with that endpoint's allowlist; the class exposes `.apply(...)` (sort/search return a filtered/sorted copy; projection mutates+returns a dict). Per-endpoint usage: build the dep classes at module load (so FastAPI sees stable types and the factory's allowlist validation catches typos before request time), then `items = search.apply(items); items = sort.apply(items); return paginate(items, pagination)` (or `count_and_page`+`page_envelope` when the page needs a per-item transform). Sort's `apply()` defaults to `uuid` as a secondary key so identical-timestamp items don't shuffle across page boundaries (sqlite `CURRENT_TIMESTAMP` is second-resolution); pass `secondary_key="id"` for entities where bulk insertion is the common path (e.g. annotation items — `id` is the autoincrement and preserves insertion order, whereas `uuid` is random and would shuffle whole batches). Search standardizes on `?q=` (no per-endpoint name override) and case-insensitive substring; blank/whitespace is a no-op so FE inputs don't need clearing logic. Disallowed `sort_by` 422s at request time via manual `HTTPException` since `Literal` can't be built dynamically. Search/sort/paginate run **post-fetch in Python** — fine at current entity scale; push down into SQL inside the handler if a table outgrows that. Adopters: `GET /annotation-tasks/{uuid}/summary` uses `PaginationParams`+sort+search at the **item level** (each item explodes into ≥1 row per `evaluator × version`, `evaluators[]`/`annotators[]` blocks stable across pages) and inlines `pagination: {total, limit, offset}` in the body (its own composite shape, not the generic envelope). The **public list endpoints** — `GET /agents`, `/tests`, `/evaluators`, `/agent-tests/agent/{uuid}/tests` (name `?q=` search), `/annotation-tasks` (name `?q=`), and `/agent-tests/agent/{uuid}/runs` (`type`/`status`/`has_failures` filters, no name search — run names are auto-generated) — return `PaginatedResponse[T]` where `T` is the endpoint's **slim list model** (see the slim-summary-models bullet below); the slim projection runs only over the returned page. The JWT-only global `GET /agent-tests/runs` (workspace-wide, `get_current_org`) gets the **same** envelope + `type`/`status`/`has_failures` filters + paging — kept in lockstep with the per-agent runs endpoint so both read `.items`/`.total` identically. Search+filters run **before** the slice so `total` and the page reflect the filtered set. For the runs endpoints the per-run display names (`Run N`/`Benchmark N`) are assigned over the full ordered set **before** status/has_failures filtering, so a filter never renumbers them. **This envelope is a shape change from the historical bare-`List[...]`/`{runs}` responses — the web frontend reads `.items`/`.total`, and both runs endpoints return `.items` (not the old `.runs`).** Adding these params/shape does NOT touch routes, so the SDK overlays need no change (but the public spec's response schema changes, so clients regenerate on deploy). Other list endpoints (`/tools`, …) still return full bare lists — adopt `PaginatedResponse[T]`+`paginate` when adding pagination; don't invent a per-router shape. **Public DETAIL endpoints trim via `?compact=true` + per-endpoint filters, not slim models.** `make_projection_params(heavy_fields=[...])` builds a `compact: bool` dep whose `.apply(model.model_dump())` **nulls** the listed heavy fields *in place* (key kept, value → `None`) — so the wire shape stays `response_model`-compatible and the generated SDK/MCP types are unchanged; the weight shed is the values (transcripts, rubrics, prompts), not the key names. **Any field a projection nulls MUST be `Optional` on its model** (e.g. `EvaluatorVersionResponse.system_prompt`, `TaskSummaryResponse.item_comments` were relaxed to `Optional` solely for this) or `response_model` validation 500s. Path syntax per heavy field: plain key `"config"`, `[]` iterates a list (`"results[].output"`), `*` iterates a dict's values (`"rows[].annotations.*.reasoning"`); missing paths are silent no-ops. Adopters (all `tags=["Public API"]`, all handlers `model_dump()` → filter → `projection.apply()` → return dict): `GET /agent-tests/run/{task_id}` (`?only_failed` keeps `results` where `passed is False`; `?compact` drops `results[].{output,test_case,judge_results,reasoning}` + `evaluators[].output_config`), `GET /agent-tests/benchmark/{task_id}` (`?only_failed` narrows each model's `test_results`; `?compact` drops `model_results[].test_results` + rubrics), `GET /annotation-tasks/{uuid}/summary` (`?disagreement_only` filters at the **item** level *before* pagination — keeps items with ≥1 row where `evaluator_agreement` is set and `< 1`, so `pagination.total` reflects the disagreeing items and paging covers the whole set — then drops the agreeing rows within each surviving item; item- and row-level filters share one predicate (`_item_disagrees` mirrors the per-row agreement calc); `?compact` drops row payloads/reasoning + evaluator version bodies + `item_comments`), `GET /evaluators/{uuid}` (`?compact` drops `versions[].{system_prompt,output_config,variables}`). Like the list params, these add no routes → overlays untouched, but the public spec changes so clients regenerate on deploy. **`?compact` descriptions are auto-generated by the factory (list the dropped fields); hand-written filter `Query` descriptions must still pass `test_api_docs_style.py` — no nullability caveats (`| null` in the type already conveys it).**
- **List endpoints return slim summary models, not the full entity.** Several list/index endpoints deliberately return a trimmed shape distinct from their detail/full model, so heavy per-row fields stay detail-only (each is the `T` inside the `PaginatedResponse[T]` envelope from the pagination bullet above): `GET /agents` → `AgentSummary` `{uuid, name, type, updated_at, connection_verified}` (the full `config`, incl. credential-bearing `agent_headers`, is detail-only via `GET /agents/{uuid}` — never ship it in a bulk list); `GET /tests` **and** `GET /agent-tests/agent/{uuid}/tests` → a shared `TestListResponse` `{uuid, name, type, created_at, updated_at, config:{description}}` (defined in [tests.py](src/routers/tests.py) via `to_test_list_response()`, imported by [agent_tests.py](src/routers/agent_tests.py); drops `evaluators[]` and `config.history/evaluation/settings` — full shape on `GET /tests/{uuid}`, and note the projection deliberately **skips** the `get_evaluators_for_test` N+1); `GET /evaluators` → `EvaluatorResponse.live_version` is the slim `EvaluatorLiveVersionSummary` `{uuid, version_number, judge_model, variables}` (drops `system_prompt`/`output_config`; full version bodies live on `GET /evaluators/{uuid}` via `versions[]`). This mirrors the pre-existing `SimulationListResponse` vs `SimulationDetailResponse` split and the agent-test run-list index (`AgentTestRunListItem` + `TestRunCaseSummary`/`ModelRunSummary`, #121). **Rules:** never assume a list response carries the full entity — the frontend edit/detail views refetch `GET /{uuid}`; keep heavy or credential-bearing fields off list responses; and when the trimmed and full models diverge, add a *separate* list model rather than narrowing the shared `AgentResponse`/`TestResponse`/`EvaluatorResponse` (which POST/PUT/detail reuse). All four trimmed endpoints are `tags=["Public API"]`, so the overlays need no edit (no route add/remove/rename) but a trim re-publishes slimmer SDK/CLI/MCP models on next publish.
- **Default prompts API** — `GET /evaluators/default-prompt?purpose={llm|llm-general|stt|tts|conversation}` returns the canonical default system prompt + suggested config for prefilling the create-evaluator form. The same prompts power the seeded LLM/STT/TTS evaluators (built via `_seed_from_purpose` in [db.py](src/db.py) so `DEFAULT_PROMPTS_BY_PURPOSE` is the single source of truth). The `llm`, `llm-general`, and `conversation` purposes' prompts are templates with a literal `<ENTER EVALUATION CRITERIA HERE>` placeholder the user replaces directly (no `{{var}}`); the `llm`/`llm-general` *seeds* (`default-llm-next-reply`, `default-llm-general`) instead carry a real `{{criteria}}` variable. For `default-llm-next-reply` (the LLM-test flow) that variable substitutes per-test via calibrate `arguments`; for `default-llm-general` it substitutes per-item via `calibrate general`'s flat per-row `arguments` (fed from each annotation item's `evaluator_variables`), the same `{{var}}` mechanism. Default judge models: `DEFAULT_TEXT_JUDGE_MODEL` for text/STT/conversation/llm-general, `DEFAULT_AUDIO_JUDGE_MODEL` for TTS.

## Conventions

- New entities: UUID primary identifier (plus auto-increment `id`), `deleted_at` soft delete, `created_at`/`updated_at` timestamps, JSON `config` column for flexible fields, validated at the API layer via Pydantic.
- Many-to-many links use dedicated pivot tables with their own soft-delete column.
- When mutating child rows (e.g. `dataset_items`), bump the parent's `updated_at` in the same transaction.
- Schema migrations on existing tables: `ALTER TABLE ADD COLUMN` inside `init_db()`, wrapped in `try/except sqlite3.OperationalError: pass`. Use `DEFAULT NULL` — SQLite silently rejects non-constant defaults like `CURRENT_TIMESTAMP` in `ADD COLUMN`, and the `except` will swallow the failure.
- All job failures route through `capture_exception_to_sentry()` (marks as unhandled, flushes immediately — critical for background threads).
- CLI failure detection is two-layer: non-zero exit code **or** expected structured output missing. Never treat stderr content as a failure signal — the calibrate CLI emits benign cleanup tracebacks.
- **API `status`/`type` fields are concretely typed via shared enums/Literals in [utils.py](src/utils.py)** — never re-loosen a request/response model field to bare `str` when a closed value set exists. Two status vocabularies exist and must not be crossed: the **run family** (`TaskStatus`: `queued/in_progress/done/failed`, `cancelled` reserved; abort lives in `details.aborted`, not `status`) covers STT/TTS eval, agent-test, simulation, and annotation-eval jobs; the **annotation family** (`AnnotationStatus`: `pending/queued/in_progress/completed/failed`) covers labelling jobs and the public annotation-eval view, which normalizes the run family's internal `done` → `completed` via `_shape_eval_job_for_response`. Typing a response `status` with the *wrong* family (or omitting a value a writer can emit) turns an egress into a 500 via response-model validation — audit every write path before narrowing. Conversely, when a field can only hold a *subset* at a given lifecycle point, type it to that subset so the public docs advertise only the reachable values: **job-create responses** (`TaskCreateResponse`, `AgentTestRunCreateResponse`, `BatchTestRun`) use `InitialTaskStatus` (a two-value `queued`/`in_progress` Literal in utils.py), never the full `TaskStatus` — the launch path only ever emits those two. **Never re-list an enum's own values in a `Field(description=...)`** — the docs renderer (Mintlify) auto-lists them from the schema, so prose states purpose only (or glosses what each value *means*); a house-style convention documented in the `api-writing-style` skill. The `type`/`role` Literals (`SimulationRunType`, `AgentTestJobType`, `EvalJobType`, `AnnotationTaskTypeLiteral`, `TestTypeLiteral`, `EvaluatorTypeLiteral`, `MemberRoleLiteral`, …) **mirror DB constants/CHECKs** (`db.ANNOTATION_TASK_TYPES`, `db.VALID_EVALUATOR_TYPES`, `CHECK(role IN …)`) — Literals can't reference runtime tuples, so keep both in sync when the vocabulary changes. Tightening a request field means invalid input now 422s at the schema layer (not a hand-rolled 400) — update the matching test's asserted status code.
